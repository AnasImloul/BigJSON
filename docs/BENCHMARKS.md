# Benchmarks: jsq vs jq

`jsq` and [`jq`](https://jqlang.github.io/jq/) solve overlapping problems, so the fair
question is: on the same file and the same query, how do they compare on **wall time**
and **memory**? The short answer: `jq` parses the whole document into the heap, so its
memory scales with file size and it slows down sharply past ~100 MB. `jsq` memory-maps
the file and streams a single pass, so its actual RAM use stays nearly flat — tens of
MB even on a 1 GB file — and it pulls ahead on time as files grow.

The harness, generator, and queries are in [`../bench`](../bench) — everything here is
reproducible.

## Setup

- **Machine:** Apple M2 Max, 32 GB RAM, macOS (Darwin 25.2), SSD.
- **Tools:** `jsq` 1.0.0 (release build), `jq` 1.7.1, `hyperfine` 1.20.
- **Data:** a single JSON object `{"events":[ … ]}` — one large array of records with
  scalar fields plus a small nested `items` array (generated by
  [`bench/gen.py`](../bench/gen.py), seeded for reproducibility).
- **Time:** `hyperfine` with 1 warmup run, reporting the **median** of 5 runs (3 at
  1 GB). Output is redirected to `/dev/null` so streaming the result doesn't skew timing.

### Two memory numbers

Both come from `/usr/bin/time -l` on a separate single run:

- **RAM** = *peak memory footprint* (`phys_footprint`). This is the dirty + compressed
  memory the process actually owns — the number **Activity Monitor shows in its
  "Memory" column**. It excludes clean, file-backed pages the kernel can drop for free.
- **RSS** = *maximum resident set size*. This also counts the memory-mapped source and
  index pages faulted in during the scan. Those pages are clean and reclaimable (the
  kernel can evict them without swapping), so RSS overstates the memory jsq truly needs.

For `jq` the two are nearly identical — it's all heap. For `jsq` they diverge sharply,
and **RAM is the honest "how much memory does this need" figure.** RSS is included for
completeness.

## Queries

Four equivalent pairs of increasing complexity. The `jq` side uses the efficient
idiomatic form — a `reduce` into an object for group-bys, not `group_by` — so jq isn't
strawmanned. **All four pairs produce identical output, verified before timing.**

| #  | What it does                            | jsq                                                |
|----|-----------------------------------------|----------------------------------------------------|
| q1 | filter rows, project two fields         | `from .events[] as e where e.amount > 500 select { … }` |
| q2 | filter rows, count them                 | `… where e.status == "paid" aggregate { n: count() }`   |
| q3 | group by region, sum a field            | `aggregate { revenue: sum(e.amount) } by e.region`      |
| q4 | group by region, 3 metrics, sorted      | `aggregate { revenue, n, avg } by e.region order by …`  |

| q2 | 4.494s | 33.378s | 7.4x |   34 MiB | 16969 MiB | 499x | 2352 MiB | 11110 MiB |
| q3 | 5.009s | 50.736s | 10.1x |  34 MiB | 16969 MiB | 499x | 2352 MiB | 10028 MiB |

At this size the two jq numbers cross over: its **RAM (17 GB) exceeds its RSS (~11 GB)**
because macOS compresses pages under memory pressure — `phys_footprint` counts the
compressed copies, while RSS only counts what's still resident uncompressed. jsq never
gets near that pressure: its aggregate queries hold a flat **34 MiB** of real memory
while scanning the full gigabyte.

## Takeaways

- **Memory is the headline.** `jq` holds the entire parsed document on the heap —
  roughly **16× the file size** here — so a 1 GB file needs ~17 GB of RAM. `jsq` keeps
  its records in a memory-mapped index, not the heap, so its actual RAM (the
  Activity-Monitor figure) stays in the tens of MB regardless of file size. At 100 MB
  that's ~33 MiB vs ~1.7 GB; at 1 GB it's still **34 MiB vs ~17 GB — a ~500× difference**
  on the aggregate queries.
- **mmap is why RSS and RAM differ.** jsq's larger RSS is mostly clean, file-backed
  pages (the source and the index) that the kernel can reclaim without swapping. That's
  also what lets jsq query files **larger than RAM** — something the parse-everything
  approach can't do at all.
- **The speed gap widens with size.** At 10 MB jsq is ~1.6–2.3× faster; by 100 MB it's
  ~4–6×, because jq's full parse and allocation cost grows with the document while
  jsq's single pass does not.
- **Group-bys (q3/q4) favor jsq most**, since jq must build and hold intermediate
  structures while jsq folds them away as it scans. The `q1` projection uses more memory
  than the aggregates on both tools because it materializes a large result stream.
- **jq wins at small scale.** Below ~5 MB, jq's lower process startup makes it faster on
  the simple queries — the whole job is over before jsq's index build pays off. Reach
  for jsq when files are large; reach for jq for quick hits on small files and its far
  richer transformation language.
